# LLM Tool Hub V2 Ideas

## 방향

V2는 단순 기능 추가 버전이 아니라, 이 허브에서 무엇을 독립 제품으로 키울지 결정하는 단계다.

사업적으로 보면 현재 구조는 하나의 "운영 코어"에 세 가지 가능성이 겹쳐 있다.

- 개인 LLM 운영 허브
- 중앙 메모리 기반 개인화 관리 시스템
- Prompt Forge 및 크롬 익스텐션 제품군의 모체

V2의 목적은 이 셋을 모두 키우는 것이 아니라, 가장 강한 축을 중심으로 재배치하는 것이다.

## V2 제품 가설

가장 유력한 V2 가설은 아래다.

### 가설 A. Prompt Forge 중심 제품화

핵심:
- 프롬프트 재사용
- 프롬프트 보관함
- 개인화된 재작성
- 웹 허브 + 크롬 익스텐션 연동

왜 유력한가:
- 사용 가치가 명확하다
- "다시 쓰는 프롬프트"는 반복 사용성이 높다
- 익스텐션과 연결될 때 사용 빈도가 올라갈 가능성이 크다

### 가설 B. Personal AI Control Panel

핵심:
- 모델/툴 비교
- 메모리/개인화 관리
- 실험용 프롬프트 워크벤치

왜 의미 있는가:
- 현재 프로젝트 구조와 가장 잘 맞는다
- 내부 운영 허브로서 완성도가 높아질 수 있다

한계:
- 대중 제품으로는 메시지가 약할 수 있다

### 가설 C. Memory OS

핵심:
- 중앙 메모리 프로필
- 장기/중기/단기 문서
- 개인화 입력과 스킬/추천 연계

왜 흥미로운가:
- 장기적으로 강한 차별점이 될 수 있다

한계:
- V2에서 바로 제품화하기엔 설명 비용이 크다
- 사용자에게 보이는 가치보다 내부 아키텍처 가치가 더 크다

## 추천 우선순위

V2에서는 아래 우선순위를 권장한다.

1. Prompt Forge 중심으로 사용자 가치 명확화
2. 중앙 메모리는 차별화 엔진으로 유지
3. 모델/툴 허브 기능은 보조 운영 패널로 정리

즉, 겉으로는 Prompt 제품이 앞에 서고, 안쪽에서는 memory와 registry가 받쳐주는 구조가 가장 현실적이다.

## 탭별 V2 확장 아이디어

### Home

아이디어:
- 최근 사용한 프롬프트, 최근 로그인 상태, 최근 개인화 변경을 묶은 운영 홈
- "오늘 가장 많이 쓰는 기능" 중심으로 홈 재배치
- 사용자별 quick action 카드 도입

사업 관점:
- 홈은 설명 페이지가 아니라 리텐션 페이지가 되어야 한다
- 매번 들어왔을 때 바로 행동하게 해야 한다

### Memory

아이디어:
- 메모리 변경 이력 보기
- 메모리 diff 요약
- 어떤 메모리가 어떤 추천/프롬프트 결과에 반영됐는지 trace 보기
- 메모리 프로필 템플릿화

사업 관점:
- Memory는 직접 판매 기능보다 "차별화 엔진"으로 쓰는 것이 좋다
- 유저가 체감하는 개인화 품질이 올라가야 메모리 기능의 존재 의미가 생긴다

### Personalization

아이디어:
- 로그인 후 개인화 상태 점검 체크리스트
- 메모/선호/스타일/금지 규칙 입력 구조화
- 개인화 강도 프리셋
- Google 외 간단한 이메일 링크 로그인 검토

사업 관점:
- 개인화 입력 피로를 줄여야 한다
- 사용자가 "설정해야 해서 귀찮다"가 아니라 "한 번 세팅하면 계속 편하다"고 느껴야 한다

### Skills

아이디어:
- 추천 스킬 자동 제안
- 현재 작업 목적 기반 스킬 필터
- 스킬 즐겨찾기
- 스킬 사용 이력/빈도 시각화

사업 관점:
- 스킬 탭은 전문가용 기능이다
- 일반 사용자에게는 "추천된 몇 개만" 잘 보여주는 방식이 더 낫다

### Prompt Tailor

아이디어:
- 결과 버전 비교
- 입력 의도별 템플릿
- 카테고리별 prompt style preset
- "짧게 / 설득형 / 분석형 / 작업지시형" 같은 후처리 버튼

사업 관점:
- Prompt Tailor는 진입 기능이 될 수 있다
- 가볍고 즉각적인 효용이 있어 신규 사용자 onboarding에 유리하다

### Prompt Forge

아이디어:
- 저장 프롬프트 라이브러리
- 태그, 검색, 즐겨찾기
- 재사용 횟수/최근 사용 기준 정렬
- 프롬프트별 적용 대상
- 웹에서 만든 프롬프트를 익스텐션에서 바로 불러오기
- 프롬프트 버전 관리
- 개인용/공개용 프롬프트 구분

사업 관점:
- 이 탭이 V2의 대표 제품 후보
- "내가 이미 만든 좋은 프롬프트를 다시 쉽게 쓴다"는 메시지가 가장 강하다
- extension과 연결되면 사용 빈도와 잠금 효과가 커진다

### Prompt Translate / Live Prompt

아이디어:
- 모델별 결과 비교
- 번역 프리셋
- 실험 결과 저장
- A/B 테스트 로그
- 실제 사용처별 출력 포맷 버튼

사업 관점:
- 이 영역은 power user retention에 좋다
- 다만 첫 제품 포지셔닝의 전면에 놓기보다, 고급 기능으로 두는 편이 낫다

## V2 구조 개편 아이디어

### 안 1. 통합 허브 유지 + Prompt Forge 전면 배치

구조:
- 현재 허브 유지
- 홈과 Prompt Forge를 전면에 배치
- Memory/Skills는 운영 보조 영역으로 유지

장점:
- 현재 코드/구조를 가장 적게 흔든다
- 운영 안정성을 유지하기 쉽다

단점:
- 제품 메시지가 다소 넓게 퍼질 수 있다

### 안 2. Prompt Forge 분리, Hub는 내부 운영용으로 축소

구조:
- 외부 공개: Prompt Forge + extension
- 내부 운영: 현재 허브 유지

장점:
- 제품 메시지가 선명하다
- 사업화와 브랜딩에 유리하다

단점:
- 운영 복잡도가 올라간다
- 동기화 포인트가 늘어난다

### 안 3. Memory Core를 API/엔진처럼 분리

구조:
- 메모리 코어는 별도 엔진
- 허브와 Prompt 제품이 그것을 사용하는 구조

장점:
- 장기적으로 가장 강한 구조

단점:
- V2 시점에는 과하다
- 내부 복잡도 대비 외부 효용이 당장 크지 않을 수 있다

## 추천안

현재는 안 1이 가장 현실적이다.

즉:
- 허브는 유지
- Prompt Forge를 핵심 가치로 전면 배치
- extension 연동을 다음 성장축으로 삼고
- Memory는 차별화 엔진으로 계속 유지

이렇게 가면 기존 자산을 버리지 않고도 제품 메시지를 더 선명하게 만들 수 있다.

## V2에서 검증하면 좋은 실험

1. 저장 프롬프트 라이브러리 사용 빈도
2. 웹 허브에서 만든 프롬프트를 익스텐션에서 다시 불러오는 빈도
3. 개인화 메모를 쓴 사용자와 안 쓴 사용자의 재방문 차이
4. Prompt Tailor를 첫 진입 기능으로 둘 때 전환율 변화
5. 공개 프롬프트 공유 기능이 실제로 쓰이는지 여부

## V2 성공 기준

아래 중 3개 이상이 보이면 V2 방향이 맞다고 본다.

1. 사용자가 프롬프트를 "새로 만드는" 것보다 "다시 꺼내 쓰는" 행동을 반복한다.
2. 개인화 설정을 한 사용자의 재사용률이 더 높다.
3. 웹 허브와 익스텐션 사이 이동이 자연스럽게 발생한다.
4. Prompt Forge를 별도 제품으로 설명했을 때 가치가 더 선명해진다.
5. Memory 기능이 사용자에게는 보이지 않아도 결과 품질 차이로 체감된다.
